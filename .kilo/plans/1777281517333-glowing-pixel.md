# Plan: Integrate IO Tasks Design System into Dashboard Theme-Editor Page

## Objective
Transform the HTML-based design system from `docs/io_tasks_design_system_full.html` into optimized React components styled with the project's default theme (CSS variables), and integrate them into the `dashboard/theme-editor` page. Keep changes isolated to this page only, avoiding global application modifications.

## Current State Analysis
- **Theme-Editor Page**: Already uses inline CSS variables for preview components (buttons, badges, inputs, cards). Supports light/dark themes with neobrutalism styling.
- **Design System Source**: HTML file with CSS classes defining components like buttons, badges, inputs, cards, pills, avatars, etc., using a dark/light theme system with CSS variables.
- **Project Structure**: Uses shadcn/ui components in `src/components/ui/`, but for isolation, we'll create new components or extend existing ones locally in the page.

## Challenges & Assumptions
- The HTML uses class-based CSS, while the project uses inline styles and CSS variables for theming.
- Need to map HTML classes to React props and style them using the existing theme variables (e.g., `--bg`, `--t1`, `--shadow-nb`).
- Isolate changes: No modifications to global CSS files or existing UI components outside this page.
- Assume the design system is complete and matches the project's neobrutalist aesthetic.

## Proposed Solution
1. **Extract Components**: Identify and categorize components from the HTML (e.g., buttons, badges, inputs, cards, pills, avatars, etc.).
2. **Transform to React**: Create local React components in the theme-editor page file, converting class-based styles to inline styles using CSS variables.
3. **Theme Integration**: Use the page's existing CSS variable system (light/dark modes) to style components.
4. **Add to Library Section**: Extend the existing component library in the theme-editor page to include the new components.
5. **Maintain Isolation**: All new code stays within `page.tsx`; no new files or global changes.

## Detailed Steps
1. **Component Extraction & Mapping**:
   - Analyze HTML sections: tokens, components, inputs, cards, sidebar, command.
   - Map to existing preview components or create new ones (e.g., pills, avatars, IO badges).
   - Convert CSS class styles to React inline styles using the page's `tokens` object.

2. **Create New Components**:
   - `PreviewPill`: For action pills (e.g., streak, progresso).
   - `PreviewIOBadge`: For IO score display.
   - `PreviewAvatar`: For user avatars.
   - Extend `PreviewCard` for compact/expanded variants if needed.
   - Add any missing variants (e.g., btn-habit, btn-task).

3. **Style Transformation**:
   - Replace HTML CSS vars (e.g., `--bg`, `--t1`) with page's equivalent (e.g., `var(--background)`, `var(--foreground)`).
   - Ensure neobrutalist effects (shadows, transforms on hover) match the page's `PreviewButton` logic.
   - Handle light/dark mode switching via the page's `isDark` state.

4. **Integration into Page**:
   - Add new sections to the component library (e.g., "Action Pills", "IO Badge & Avatar").
   - Use the existing `Section` and `Row` helpers.
   - Update the preview area to showcase all components.

5. **Testing & Validation**:
   - Ensure components render correctly in both themes.
   - Verify hover effects and interactions work.
   - Check responsiveness and layout.

## Risks & Mitigations
- **Complexity**: HTML has many components; prioritize core ones (buttons, badges, inputs, cards) first.
- **Styling Conflicts**: Use page-specific CSS vars to avoid conflicts.
- **Performance**: Keep components lightweight; avoid heavy computations.
- **Isolation**: Double-check no imports from outside the page.
